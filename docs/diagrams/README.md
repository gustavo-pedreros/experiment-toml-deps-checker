# Architecture diagrams

Four hand-drawn SVG diagrams covering the project's structure at
different zoom levels. Open any file in a browser or render in
Markdown viewers that support inline SVG. All diagrams are
generated by hand (no `dot` / `mmdc` / `d2` dependency); when the
underlying structure shifts, edit the SVG directly.

## Conventions

- **Purple** = `application` (ports + use cases)
- **Green** = `infrastructure` (adapters)
- **Lavender / purple-mixed** = `checks` (rule engines)
- **Yellow / amber** = `presentation` (CLI commands, Rich console)
- **Rose** = `bootstrap` / `cli` (composition root)
- **Blue-grey** = `domain` (pure value objects, deepest layer)
- **Solid green arrows** = allowed import direction
- **Dashed red arrows** = forbidden import direction (enforced by `import-linter`)
- **Orange** = external HTTP services (Maven Central, etc.)

## Regenerating

These SVGs are source-of-truth artefacts — no build step. To update,
edit the `.svg` file directly with a text editor or a vector editor
(Inkscape / Boxy / Figma → export SVG). When the architecture
itself shifts (a new RFC adds a layer, a port is split, a new
external service is wired in), update the corresponding diagram
in the same PR for traceability.

The hand-drawn approach is deliberate: it sidesteps the
`mermaid` / `dot` / `d2` toolchain dependency and keeps the
visual exactly as documented (no surprise relayouts on the next
renderer release). Trade-off: edits cost a few minutes of manual
SVG tweaking.
